# 命令与模板改造指南

## 概述

现有SDD命令集以线性流水线方式运行，没有正式的门径机制。本指南规定了每个命令和模板如何增强以融入IPD技术评审（TR，Technology Review）门径感知能力，同时保留原始功能和向后兼容性。

集成采用"门径检查点"模式：每个命令增加一个前置门径验证步骤，检查当前阶段所需的TR标准是否已满足。如果未满足，命令将警告用户并可选择阻止执行。

## TR门径映射汇总

| TR门径 | 阶段 | 适用命令 |
|--------|------|----------|
| TR0 | 项目初始化 | `/speckit-constitution` |
| TR1 | 概念 | `/speckit-specify`、`/speckit-clarify` |
| TR2/TR3 | 计划/设计 | `/speckit-checklist`、`/speckit-plan` |
| TR4 | 开发准入 | `/speckit-tasks` |
| TR4/TR4A | 开发执行 | `/speckit-implement` |
| TR5 | 验证 | `/speckit-analyze` |

## 命令改造详情

### `/speckit-constitution`

**当前行为**：从用户原则输入创建/更新项目宪法。没有门径验证。

**IPD门径集成**：增加TR0门径验证，检查项目是否已有已批准的宪法再允许更新。首次创建时，自动将批准日期注册为TR0门径批准日期。

**新行为**：
- 首次运行：创建宪法 + 注册TR0门径批准
- 后续运行：检查拟议变更是否与现有TR门径兼容（不得无重新评审地追溯删除原则）
- **TR检查点**：TR0（项目初始化）

### `/speckit-specify`

**当前行为**：从用户描述创建功能规格。没有门径验证。

**IPD门径集成**：增加TR1门径感知。创建新规格前，验证项目宪法已批准（TR0已通过）。规格创建后，生成TR1就绪清单，显示哪些概念阶段标准已满足。

**新行为**：
- 前置检查：检查TR0状态（宪法必须存在）
- 创建后：生成TR1就绪报告（规格完整性、可行性指标、初始风险评估）
- **TR检查点**：TR1（概念）

### `/speckit-clarify`

**当前行为**：通过问答减少规格模糊性。没有门径验证。

**IPD门径集成**：所有澄清解决后，生成风险评估输出，作为TR1门径决策的输入。追踪哪些模糊性已解决、哪些风险仍开放。

**新行为**：
- 会话中：为每个澄清标记风险级别（高/中/低）
- 会话后：生成TR1风险摘要作为门径证据
- **TR检查点**：TR1（概念）

### `/speckit-checklist`

**当前行为**：生成需求验证质量检查清单。没有门径验证。

**IPD门径集成**：重新定位为TR2准入门径验证。检查清单输出成为TR2（架构/计划就绪）的门径证据。

**新行为**：
- 前置检查：检查TR1门径状态（规格必须已批准）
- 生成后：检查清单完整性分数作为TR2准入标准
- **TR检查点**：TR2（架构就绪）

### `/speckit-plan`

**当前行为**：从规格和用户技术栈输入创建技术实施计划。没有门径验证。

**IPD门径集成**：增加TR2/TR3门径检查点。计划创建前，验证TR1已通过。计划创建后，生成架构决策日志（ADL）作为TR3门径证据。

**新行为**：
- 前置检查：检查TR1门径状态
- 创建后：生成架构决策日志作为TR3输入
- 包含宪法检查章节（计划模板中已有）
- **TR检查点**：TR2（架构评审）/ TR3（设计批准）

### `/speckit-tasks`

**当前行为**：将计划分解为可执行任务。没有门径验证。

**IPD门径集成**：增加TR4门径检查点。生成任务前，验证架构/设计评审（TR2/TR3）已通过。任务成为TR4/TR4A期间跟踪的开发基线。

**新行为**：
- 前置检查：检查TR2/TR3门径状态
- 生成后：任务数量和工作量估算作为TR4基线
- **TR检查点**：TR4（开发基线）

### `/speckit-implement`

**当前行为**：执行实施任务。没有门径验证。

**IPD门径集成**：增加TR4/TR4A门径检查点。实施前验证TR4已通过。实施后生成质量报告（测试覆盖率、静态分析、安全扫描）作为TR4A门径证据。

**新行为**：
- 前置检查：检查TR4门径状态
- 持续收集：执行期间收集质量指标
- 完成后：生成TR4A质量报告用于灰度发布就绪评估
- **TR检查点**：TR4（开发）/ TR4A（灰度发布就绪）

### `/speckit-analyze`

**当前行为**：跨产物一致性分析。没有门径验证。

**IPD门径集成**：重新定位为TR5验证门径。分析输出成为TR5（系统验证）门径评审的主要证据。

**新行为**：
- 前置检查：检查TR4A门径状态（实施必须完成）
- 分析范围：扩展到包括IPD门径合规、DoD验证和Must-Meet标准覆盖
- 分析后：生成TR5验证报告
- **TR检查点**：TR5（系统验证）

## 模板改造详情

### 宪法模板

**当前章节**：核心原则（5项）、附加章节（2项）、治理

**新增IPD对齐章节**：
- 增加"敏捷-门径治理"作为独立原则章节
- 增加"工具与平台要求"章节
- 增加"门径标准参考"章节，列出每个TR门径的Must-Meet/Should-Meet定义
- 治理：增加修订程序，要求MAJOR版本变更时重新进行TR门径评审

**向后兼容**：是 — 新章节为增量添加；现有宪法仍然有效。

### 规格模板

**当前章节**：用户故事、需求、关键实体、成功标准、假设

**新增IPD对齐章节**：
- 增加"TR门径评估"章节，列出适用哪些TR门径及所需证据
- 增加"风险登记"章节用于初始风险识别
- 成功标准：增加Must-Meet/Should-Meet分类

**向后兼容**：是 — 新章节为增量添加。

### 计划模板

**当前章节**：摘要、技术上下文、宪法检查、项目结构、复杂度追踪

**新增IPD对齐章节**：
- 增加"门径就绪"章节，替代/前置宪法检查
- 增加"TR门径证据清单"章节
- 增加"WSJF优先级分数"章节用于待办列表优先级排序

**向后兼容**：是 — 门径章节仅调整宪法检查的位置；检查本身保留。

### 任务模板

**当前章节**：按用户故事分阶段的任务、依赖关系、执行顺序

**新增IPD对齐章节**：
- 在每个阶段检查点增加"门径完成验证"
- 增加"DoD检查清单模板"作为附录
- 增加"质量门径标准"用于每个阶段转换

**向后兼容**：是 — 门径章节为增量注释。

## 实施顺序

推荐实施这些变更的顺序：

1. **宪法模板先** — 建立门径框架
2. **规格模板** — 为新功能启用TR1门径
3. **计划模板 + `/speckit-plan`** — 启用TR2/TR3
4. **任务模板 + `/speckit-tasks`** — 启用TR4
5. **`/speckit-implement`** — 启用TR4/TR4A
6. **`/speckit-analyze`** — 启用TR5
7. **`/speckit-constitution`、`/speckit-specify`、`/speckit-clarify`、`/speckit-checklist`** — 其余命令门径集成

此顺序确保每个门径在实施时可端到端测试。

## 向后兼容说明

- 所有模板变更为增量添加 — 现有文档仍然有效
- 命令增加前置检查，但在没有门径上下文时保留现有行为（如未配置TR门径的项目）
- TR门径系统可以按项目逐步引入
