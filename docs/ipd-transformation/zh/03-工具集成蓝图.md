# 工具集成蓝图

## 概述

敏捷-门径混合模型需要将IPD技术评审（TR, Technology Review）门径强制执行集成到开发工作流中。本蓝图描述vibe-ipd原生的**文档状态模式**——通过AI代理命令和项目文档结构实现门径强制执行，无需任何外部工具或平台配置。

vibe-ipd工具链将TR门径感知直接嵌入到每个`/vipd-speckit-*`命令中。当IPD模式激活时（通过检测项目宪法中是否存在"门径标准参考"/Gate Criteria Reference章节来判断），命令自动对项目自身文档（宪法、规格、计划、任务）执行深度内容验证的前置门径检查。

## vibe-ipd代理集成（文档状态模式）

vibe-ipd工具链通过其AI代理命令系统提供原生门径强制执行能力。这种"文档状态模式"无需外部工具即可工作，通过让每个`/vipd-speckit-*`命令对项目自身的文档执行前置TR门径检查来实现。

### 工作原理

1. **IPD模式检测**：每个命令首先检查项目宪法是否包含"门径标准参考"（Gate Criteria Reference）章节。如果包含，则IPD模式激活；如果不包含，项目为仅SDD模式，跳过门径检查。

2. **深度内容验证**：不同于简单的文件存在性检查，每个命令执行深度内容验证——验证相关文档（宪法、规格、计划、任务）中存在所需的章节内容。

3. **门径流程强制执行**：命令按顺序执行门径进度检查：
   - `/vipd-speckit-constitution` → TR0（项目初始化）
   - `/vipd-speckit-specify`、`/vipd-speckit-clarify` → TR1（概念）
   - `/vipd-speckit-checklist`、`/vipd-speckit-plan` → TR2/TR3（计划/设计）
   - `/vipd-speckit-tasks` → TR4（开发基线）
   - `/vipd-speckit-implement` → TR4/TR4A（开发）
   - `/vipd-speckit-analyze` → TR5（验证）

### 对比：文档状态模式 vs 外部工具集成

| 方面 | 文档状态模式 | 外部工具集成 |
|------|-------------|-------------|
| 所需外部工具 | 无 | 取决于平台选择 |
| 门径强制执行 | 前置命令检查 | 平台自动化规则 |
| 证据收集 | 文档内容分析 | 平台问题字段 |
| 依赖追踪 | 交叉引用链接 | 平台依赖视图 |
| 设置工作量 | 无（内置于vibe-ipd） | 需要平台配置 |

## 平台替代方案

对于需要与外部项目管理平台集成的团队，门径流程保持不变；只有自动化机制不同。通用模式（TR门径定义、Must-Meet/Should-Meet标准、深度内容验证）适用于任何平台。IPD/Agile术语的规范定义请参考[术语表](glossary.md)。
