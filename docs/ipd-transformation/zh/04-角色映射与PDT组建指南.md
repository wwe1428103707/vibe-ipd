# 角色映射与PDT组建指南

## 概述

IPD-敏捷混合模型需要清晰的角色定义，以桥接IPD的宏观治理职责和敏捷的自组织团队结构。本指南将每个IPD角色映射到其敏捷/SAFe对应角色，并定义产品开发团队（PDT，Product Development Team）在敏捷-门径工作流中的运作方式。

## IPD到敏捷角色映射表

| IPD角色 | 敏捷/SAFe角色 | 核心职责 | 问责对象 |
|---------|-------------|---------|----------|
| **PDT经理（LPDT）** | 敏捷发布火车工程师（RTE） | 跨团队协调、依赖解决、TR门径主持、障碍清除 | 交付承诺、门径就绪 |
| **产品经理** | 产品负责人（PO） | 待办列表排期、用户故事定义、干系人沟通、验收标准签署 | 功能价值交付 |
| **研发代表/开发负责人** | 系统架构师/技术负责人 | 架构决策、CBB管理、技术标准、技术刺针监督 | 技术质量、架构合规 |
| **测试代表/测试负责人** | QA主管 | 测试策略、DoD定义、质量指标、环境管理 | 发布质量、测试覆盖 |
| **运维代表/运维负责人** | DevOps主管 | CI/CD流水线、部署自动化、监控、事件响应 | 运维稳定性 |

### 小团队角色组合

对于3–5人的团队，角色可以合并：

| 团队规模 | 推荐组合 |
|---------|---------|
| 3人 | LPDT + PO（1人）、研发 + 运维（1人）、测试 + UX（1人） |
| 4人 | LPDT（1人）、PO + UX（1人）、研发 + 架构师（1人）、QA + 运维（1人） |
| 5人 | 完整PDT，其中UX资源共享 |

## 产品三人组

产品三人组（Product Trio，产品三人组）是探索轨的引擎，在功能进入交付待办列表前进行验证。

### 组成

| 角色 | 人员 | 贡献 |
|------|------|------|
| **产品负责人（PO）** | 产品经理 | 市场/用户需求、商业论证、成功标准 |
| **系统架构师** | 研发负责人 | 技术可行性、架构约束、工作量估算 |
| **UX设计师** | 专职或共享 | 用户研究、原型、可用性验证 |

### 运作节奏

| 频率 | 活动 | 产出 |
|------|------|------|
| **每周同步**（1小时） | 评审进行中的探索、对齐优先级 | 更新的探索待办列表 |
| **每个迭代**（迭代计划前） | 将已验证功能交接至交付轨 | 带验收标准的功能简报 |
| **每个门径**（TR1/TR2） | 在门径评审中呈现发现 | 风险评估、可行性报告 |

### 权限范围

- **决策**：功能有效性（进入交付待办列表的通过/不通过）
- **建议**：相对其他功能的优先级
- **建议**：架构指南内的技术方案
- **上报**：资源限制或冲突上报给PDT经理

### 产品三人组审批要求

功能如果满足以下**任意**标准，**必须**通过产品三人组才能进入交付待办列表：
- 新功能（非现有功能的增强）
- 跨团队影响（需要2+团队协调）
- 新技术或外部依赖
- 面向用户的工作流变更

Bug修复、小幅增强和技术债务项可以在PDT经理批准后直接进入待办列表，无需通过产品三人组。

## 功能团队自主性

| 活动 | PDT经理（LPDT/RTE） | 功能团队 |
|------|---------------------|---------|
| 迭代目标设定 | 对齐路线图、批准范围 | 在范围内定义具体目标 |
| 实施方案 | 不参与（信任团队） | 完全自组织 |
| 技术选择 | 批准架构决策 | 推荐、实施 |
| 跨团队依赖 | 协调、追踪、上报 | 尽早标记阻塞 |
| 任务分解 | 高层审阅 | 自组织 |
| 日常执行 | 不参与（按需参加站会） | 自管理 |
| 质量标准 | 定义Must-Meet标准 | 负责质量执行 |
| 门径评审 | 主持、呈现 | 参加、提供证据 |

### 决策上报路径

```text
功能团队决策 → PDT经理（LPDT）上报
    → 架构委员会（如涉及跨系统影响）
        → IPMT（如涉及预算/范围变更）
```

## RACI矩阵

| 活动 | LPDT (RTE) | PO | 架构师 | QA主管 | DevOps主管 |
|------|-----------|-----|--------|--------|-----------|
| 需求定义 | I | **R/A** | C | C | I |
| 架构设计 | I | C | **R/A** | C | C |
| 开发实施 | I | I | C | I | C |
| 测试 | I | C | I | **R/A** | C |
| 部署 | A | I | I | C | **R/A** |
| 门径评审证据 | **R/A** | C | C | C | C |
| 待办列表排期 | C | **R/A** | I | I | I |
| 依赖管理 | **R/A** | C | C | I | I |
| 质量指标 | A | I | I | **R/A** | C |
| 生产监控 | I | I | C | C | **R/A** |

**R** = 负责人（执行者），**A** = 问责人（决策者），**C** = 被咨询（需要输入），**I** = 被告知（事后通知）

## 团队规模指南

- **最小可行PDT**：3人（合并角色）
- **推荐PDT**：5–9人（完整角色覆盖）
- **多PDT**：产品需要10+人时，拆分为多个PDT，按子系统或功能领域对齐，每个PDT有自己的LPDT
- **内源模型**：小组织（< 15人）可使用共享专家（一个QA、一个运维）服务多个功能团队

## 小团队适应方案

对于3–5人、一人兼顾多个IPD角色的团队：

1. **接受较慢的门径吞吐量** — 人少意味着更多上下文切换；计划门径评审时预留更长前置时间
2. **外部化专业角色** — QA、UX和运维可以作为共享服务而非PDT专职成员
3. **降低门径正式程度** — 小团队可使用轻量级TR门径（30分钟评审而非完整演示）
4. **尽可能自动化** — 投资CI/CD自动化以补偿有限的QA和运维手动容量
