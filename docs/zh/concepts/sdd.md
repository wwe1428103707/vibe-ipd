# 什么是规范驱动开发？

规范驱动开发（Spec-Driven Development）**彻底颠覆**了传统软件开发。几十年来，代码一直是王者 — 规范只是我们在编码"真正工作"开始前搭建又丢弃的脚手架。规范驱动开发改变了这一点：**规范变得可执行**，直接生成可工作的实现，而不仅仅是指导实现。

## 核心哲学

规范驱动开发是一个强调以下方面的结构化过程：

- **意图驱动开发** — 规范在定义"怎么做"之前先定义"做什么"
- **丰富的规范创建** — 使用护栏和组织原则
- **多步骤优化** — 而非从提示一次性生成代码
- **高度依赖** — 先进 AI 模型能力的规范解读

## 开发阶段

| 阶段 | 焦点 | 关键活动 |
| --- | --- | --- |
| **从零到一的开发**（"绿地项目"） | 从头生成 | 从高层需求开始；生成规范；规划实施步骤；构建生产就绪的应用 |
| **创造性探索** | 并行实现 | 探索多样化解决方案；支持多种技术栈和架构；试验 UX 模式 |
| **迭代增强**（"棕地项目"） | 棕地现代化 | 迭代添加特性；现代化遗留系统；适应流程 |

## 实验目标

我们的研究和实验聚焦于：

### 技术无关性

- 使用多样化技术栈创建应用
- 验证规范驱动开发是一种不依赖于特定技术、编程语言或框架的过程

### 企业约束

- 展示关键任务应用的开发
- 融入组织约束（云提供商、技术栈、工程实践）
- 支持企业设计系统和合规要求

### 以用户为中心的开发

- 为不同用户群体和偏好构建应用
- 支持各种开发方法（从 vibe-coding 到 AI-Native 开发）

### 创造性与迭代过程

- 验证并行实现探索的概念
- 提供健壮的迭代特性开发工作流
- 扩展流程以处理升级和现代化任务

# 什么是集成产品开发（IPD）？

集成产品开发（Integrated Product Development，IPD）是一种以产品为中心的综合开发管理框架，起源于华为引入的 IBM IPD 实践。IPD 将产品开发视为一项**投资行为**，通过分阶段的门禁评审（TR0–TR6）来管理技术风险，确保产品从概念到交付的全流程可控。

## 核心原则

| 原则 | 说明 |
|------|------|
| **产品开发是投资行为** | 每个特性/项目都需要经过商业论证，设有明确的 Go/Kill 决策点 |
| **基于市场的创新** | 需求来源于市场分析而非技术驱动，确保做正确的事 |
| **跨职能协同** | PDT（产品开发团队）由市场、开发、测试、运维等角色组成，打破部门墙 |
| **并行开发** | 发现轨道与交付轨道并行运行，缩短上市时间 |
| **结构化流程** | 明确阶段划分、门禁标准、角色职责，降低不确定性 |

## IPD 阶段门禁模型（TR0–TR6）

vibe-ipd 实现了完整的技术评审（Technical Review）门禁模型：

| 门禁 | 阶段 | 评审焦点 |
|------|------|----------|
| **TR0** | 概念启动 | 需求可行性评估，商业论证就绪 |
| **TR1** | 规范评审 | 特性规范完整性，用户故事清晰度 |
| **TR2** | 计划评审 | 实施计划完整性，架构方案可行性 |
| **TR3** | 设计评审 | 详细设计质量，接口契约定义 |
| **TR4** | 实施验证 | 代码实现完整性，测试覆盖率 |
| **TR5** | 系统验证 | 集成测试，性能/安全评估 |
| **TR6** | 发布决策 | 发布就绪检查，Go/Kill/Hold/Recycle 决策 |

每个 `/vipd-*` 命令在执行前都会对前置门禁标准进行深层内容校验，确保不跳过必要的质量关卡。

## 产品开发团队（PDT）与 RACI 矩阵

IPD 强调跨职能团队协作，vibe-ipd 将 IPD 角色映射到敏捷团队角色：

| IPD 角色 | 敏捷角色映射 | 关键职责 |
|----------|-------------|----------|
| LPDT（产品开发团队 Leader） | RTE（发布火车工程师） | 流程管理、门禁推进 |
| 产品经理 | PO（产品负责人） | 需求定义、价值排序 |
| 开发主管 | 系统架构师 | 技术方案、架构决策 |
| 测试主管 | QA 主管 | 质量保障、测试策略 |
| 运维主管 | DevOps 主管 | 部署运维、环境管理 |

## 产品三驾马车

IPD 发现轨道的核心是**产品三驾马车**（Product Trio）：

- **产品经理（PO）** — 定义"做什么"，关注用户价值和业务目标
- **系统架构师** — 定义"怎么做"，关注技术方案和系统质量
- **UX 设计师** — 定义"长什么样"，关注用户体验和交互设计

三者并行协作，在迭代开始前完成需求澄清、技术评估和设计验证，减少交付轨道中的返工。

# 什么是敏捷-阶段门禁混合模型？

敏捷-阶段门禁混合模型（Agile-Stage-Gate Hybrid）将**敏捷的灵活迭代**与**阶段门禁的严格治理**相结合，是 vibe-ipd 的核心运行模式。

## 为什么需要混合模型？

| 纯敏捷 | 纯阶段门禁 | 混合模型（vibe-ipd） |
|--------|-----------|-------------------|
| 缺乏高层治理，容易偏离商业目标 | 流程僵化，响应变化慢 | 灵活迭代 + 阶段把关 |
| 门禁决策依赖人工评审会议 | 门禁变相成为行政负担 | AI 自动深层校验，减负提效 |
| 大型项目容易失控 | 不适合快速原型探索 | 发现轨道（敏捷）+ 交付轨道（门禁） |

## 双轨运行模式

```
时间轴 →
──────────────────────────────────────────────────
发现轨道（敏捷） ── 产品三驾马车持续探索
  │  ┌──────┐  ┌──────┐  ┌──────┐
  │  │ Sprint 1 │  │ Sprint 2 │  │ Sprint 3 │  ...
  │  └──────┘  └──────┘  └──────┘
  │       │           │           │
交付轨道（门禁） ── 门禁验证驱动的实施
  │       ▼           ▼           ▼
  │  ┌─ TR0 ─┐  ┌─ TR2 ─┐  ┌─ TR4 ─┐
  │  │ 概念   │  │ 计划   │  │ 实施   │
  │  └───────┘  └───────┘  └───────┘
```

- **发现轨道**：采用敏捷 Scrum/Kanban 方式，由产品三驾马车持续探索需求、验证假设
- **交付轨道**：以阶段门禁为里程碑，每个 TR 门禁自动验证前置条件，确保质量关卡
- **两轨并行**：发现轨道的输出直接流入交付轨道的下一个门禁阶段，缩短等待时间

## 门禁决策机制

每个 TR 门禁评审产出的四种决策：

| 决策 | 含义 | 后续行动 |
|------|------|----------|
| **Go** | 通过 | 进入下一阶段 |
| **Kill** | 终止 | 停止该特性/项目开发 |
| **Hold** | 暂缓 | 暂停当前工作，等待条件成熟 |
| **Recycle** | 返工 | 返回上一阶段，修复问题后重新评审 |

## AI 增强的门禁执行

传统阶段门禁依赖人工评审和文档审核，效率低下。vibe-ipd 通过 AI 智能体实现：

- **深层内容校验** — 不仅检查文档是否存在，更理解内容是否满足门禁标准
- **自动质量门禁** — 每个 `/vipd-*` 命令自动校验前置门禁，无需手动发起评审
- **智能体编排** — 50+ 专业智能体角色协同工作，覆盖架构、安全、测试等维度
- **ADL（架构决策日志）** — 自动记录关键技术决策及其理由，支持追溯审计
