# Feature Specification: PDT Role Mapping & Agent Setup

**Feature Branch**: `007-pdt-role-setup`

**Created**: 2026-06-06

**Status**: Draft

**Input**: User description: "Implement PDT role mapping guide for Claude Code — create agent skills for each IPD role, set up Product Trio patterns, adapt RACI for AI-human collaboration"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Configure IPD Roles as Claude Code Subagents (Priority: P1)

As a solo developer using the IPD toolkit, I want each IPD role (LPDT/RTE, PO, Architect, QA Lead, DevOps Lead) to be invocable as a Claude Code subagent via `vipd-agent-assign-*` skills, so that I can get role-specific reviews and gate assessments without context-switching.

**Why this priority**: Without role-specific agents, gate reviews and Product Trio discovery lack the cross-functional perspective that IPD requires. This is the foundation for all downstream IPD workflow.

**Independent Test**: Can invoke `/vipd-agent-assign-lpdt` with a gate review prompt and receive a response framed from the LPDT/RTE perspective.

**Acceptance Scenarios**:
1. **Given** the project has `.claude/skills/vipd-agent-assign-lpdt/`, **When** I invoke `/vipd-agent-assign-lpdt`, **Then** the agent adopts LPDT/RTE responsibilities (delivery commitment, gate readiness).
2. **Given** the project has `.claude/skills/vipd-agent-assign-po/`, **When** I invoke `/vipd-agent-assign-po`, **Then** the agent adopts PO responsibilities (backlog prioritization, acceptance criteria).
3. **Given** the project has `.claude/skills/vipd-agent-assign-architect/`, **When** I invoke `/vipd-agent-assign-architect`, **Then** the agent adopts Architect responsibilities (technical feasibility, architecture compliance).
4. **Given** the project has `.claude/skills/vipd-agent-assign-qa-lead/`, **When** I invoke `/vipd-agent-assign-qa-lead`, **Then** the agent adopts QA Lead responsibilities (test strategy, quality metrics).
5. **Given** the project has `.claude/skills/vipd-agent-assign-devops-lead/`, **When** I invoke `/vipd-agent-assign-devops-lead`, **Then** the agent adopts DevOps Lead responsibilities (CI/CD, operational stability).

---

### User Story 2 - Product Trio Discovery for Feature Validation (Priority: P2)

As the Product Trio (PO + Architect + UX), I want to run a structured discovery session for a proposed feature before it enters the delivery backlog, so that desirability, feasibility, and viability are validated upfront.

**Why this priority**: Dual-Track Agile requires discovery track gatekeeping. This story enables the Product Trio workflow but depends on role agents existing (Story 1).

**Independent Test**: Can invoke `/vipd-specify` with a feature idea, then trigger a Product Trio review via assigned agents.

**Acceptance Scenarios**:
1. **Given** a feature spec exists, **When** I run the Product Trio discovery workflow, **Then** each trio member (PO, Architect, UX) provides their assessment.
2. **Given** all trio members have reviewed, **When** any member flags a blocker, **Then** the feature is marked "discovery blocked" with rationale.
3. **Given** the trio assessment is complete, **When** no blockers exist, **Then** the feature is cleared for the delivery backlog.

---

### User Story 3 - RACI-Guided Task Assignment (Priority: P3)

As a PDT manager, I want RACI annotations embedded in task descriptions so that task ownership, accountability, and consultation requirements are explicit during implementation.

**Why this priority**: RACI clarity reduces handoff delays and confusion, but is a refinement layer on top of the basic role setup.

**Independent Test**: A task generated by `/vipd-tasks` includes a `RACI:` field showing responsible and accountable roles.

**Acceptance Scenarios**:
1. **Given** a plan is broken into tasks, **When** the tasks are generated, **Then** each task includes a `RACI: R=<role>, A=<role>` annotation.
2. **Given** a cross-team dependency task, **When** the task is assigned, **Then** consulted (C) and informed (I) roles are listed.

---

### Edge Cases

- What happens when a single person fills multiple IPD roles (small team)? Agents should be able to adopt combined personas (e.g., LPDT+PO).
- What if a Product Trio member is unavailable? The process should allow async reviews with a timeout.
- How does the system handle conflicting assessments between trio members? Escalation to PDT manager (LPDT) for tie-breaking.

## TR Gate Assessment *(IPD only)*

- **Applicable TR Gates**: TR1 (Concept — Product Trio discovery gate)
- **Risk Level**: Low
- **Gate Evidence Required**: Role skill files created and functional; Product Trio review demonstrated against a sample feature

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST provide a `vipd-agent-assign-lpdt` skill that frames responses from LPDT/RTE perspective with authority over delivery commitment and gate readiness.
- **FR-002**: System MUST provide a `vipd-agent-assign-po` skill that frames responses from PO perspective with authority over backlog prioritization and acceptance criteria sign-off.
- **FR-003**: System MUST provide a `vipd-agent-assign-architect` skill that frames responses from System Architect perspective with authority over technical feasibility and architecture compliance.
- **FR-004**: System MUST provide a `vipd-agent-assign-qa-lead` skill that frames responses from QA Lead perspective with authority over test strategy and quality metrics.
- **FR-005**: System MUST provide a `vipd-agent-assign-devops-lead` skill that frames responses from DevOps Lead perspective with authority over CI/CD and operational stability.
- **FR-006**: Product Trio workflow MUST include PO, Architect, and UX Designer roles conducting parallel discovery-track review of a feature spec.
- **FR-007**: Product Trio MUST produce a go/no-go recommendation per feature before it enters the delivery backlog.
- **FR-008**: Task descriptions MUST include RACI annotations (R=Responsible, A=Accountable) for role clarity.
- **FR-009**: For small teams (3-5), combined role agents MUST be supported (e.g., LPDT+PO, Dev+Ops combined persona).

### Key Entities

- **IPD Role**: A named function within the PDT (LPDT, PO, Architect, QA Lead, DevOps Lead) with defined responsibilities and authority scope.
- **Skill File**: A `.claude/skills/vipd-agent-assign-*/SKILL.md` file that configures a Claude Code subagent to adopt a specific IPD role.
- **Product Trio**: The discovery-track decision body (PO + Architect + UX Designer) that validates features before delivery.
- **RACI Entry**: An annotation on tasks specifying Responsible, Accountable, Consulted, and Informed roles.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: All 5 IPD role skill files are creatable and invocable via `/vipd-agent-assign-*` commands.
- **SC-002**: Product Trio assessment can be completed for any feature spec in under 15 minutes (asynchronous).
- **SC-003**: Tasks generated by `/vipd-tasks` include RACI annotations with 100% coverage for Responsible and Accountable fields.
- **SC-004**: Combined role agent (e.g., LPDT+PO) produces coherent responses covering both role perspectives.

## Assumptions

- The user works solo or in a small team (1-5 people), so combined roles are the common case.
- Claude Code's subagent spawning mechanism (`Agent` tool with `subagent_type` or `name`) is used to realize multi-role reviews.
- Existing `vipd-agent-assign`, `vipd-agent-execute`, and `vipd-agent-validate` skills provide the framework; this feature adds role-specific configuration on top.
- UX Designer role is covered by general-purpose Claude capabilities rather than a dedicated skill file.

## Risk Register *(IPD only)*

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Role agents give generic advice not specific to project context | M | M | Each role skill loads `.specify/memory/constitution.md` and active spec for context |
