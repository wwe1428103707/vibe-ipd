# Architecture Decision Log: Multilingual Greeting

**Feature**: [spec.md](spec.md) | **Date**: 2026-06-07 | **Phase**: TR2/TR3 Plan & Design

## ADL-001: Embedded String Templates Over External i18n

**Status**: ✅ Accepted

**Context**: The greeting feature requires localized strings for 3 languages (en, zh, ja). Options ranged from embedded case-statements to full i18n frameworks.

**Decision**: Embed greeting templates directly in the bash script using `case` statements.

**Rationale**:
- Only 3 languages — no external i18n library justified
- Zero additional dependencies — keeps the sample project self-contained
- Pure bash: works in any POSIX shell without gettext/msgfmt

**Consequences**:
- Adding a new language requires editing the script directly
- UTF-8 terminal is required for CJK character display

## ADL-002: `--lang` Flag Over Environment Variable

**Status**: ✅ Accepted

**Context**: The language selection mechanism needed a choice between CLI flag and environment variable (e.g., `LANG`).

**Decision**: Use explicit `--lang <CODE>` CLI flag.

**Rationale**:
- Explicit > implicit — easier to test and document
- Avoids collision with system `LANG` env var
- Consistent with existing `--name`, `--help`, `--version` flag conventions

**Consequences**:
- Every invocation that needs non-English output must pass `--lang`
- Simple `case` dispatch with default fallback to English

## ADL-003: English Default for Unsupported Language Codes

**Status**: ✅ Accepted

**Context**: Behavior when user provides an unsupported language code (e.g., `--lang fr`).

**Decision**: Fall back to English with a stderr warning notice.

**Rationale**:
- Graceful degradation — user always gets output they can understand
- Non-zero exit would break scripting pipelines
- Stderr notice informs the user without polluting stdout

**Consequences**:
- Exit code remains 0 even for invalid lang codes (matching default behavior)
- Warning text must be documented in the CLI contract
