# Feature Specification: Auto Gate Fix & Report

**Feature Branch**: `019-auto-gate-fix`

**创建日期**: 2026-06-09

**状态**: Draft

**用户描述**: 现在门控未通过，根据原因由agents自动解决，不再中断。同时记录各个门控的内容。并最终生成一份报告(以静态html网页的形式)供用户查看详细内容。

## 澄清记录

### 会话 2026-06-09

- Q: HTML 报告生成命令名称？ → A: `/vipd-report`，可扩展为其他报告类型
- Q: 门控记录存储方式？ → A: 独立记录文件 `gate-history.jsonl`，纯追加写入，每行一条完整 JSON
- Q: 自动修复策略？ → A: Agent 自主判断优先（分析失败原因自行决定修复方案），门控脚本提供的 fix_hint 作为参考辅助
- Q: 安全合规门控识别方式？ → A: 门控脚本在条件结果中标记 category 字段（优先级高），Agent 关键词判断作为兜底

## 用户场景与测试 *(必填)*

### 用户场景 1 — 门控失败自动修复 (P1)

作为一名开发者，我希望当某个 TR 门控检查未通过时，系统不中断我的工作流，而是自动分析失败原因并由 Agent 尝试修复，这样我就无需手动处理每个门控的"是否继续"提示。

**为什么是这个优先级**: 这是最核心的用户痛点 — 当前门控失败会中断流程并询问用户，破坏了自动化工作流的连续性。

**独立测试**: 故意使某个门控检查失败（如删除 plan.md 后运行 pre-plan 门控），验证系统是否自动修复而非询问用户。

**验收场景**:

1. **Given** 一个缺少 plan.md 的 feature 目录，**When** 执行 `vipd-plan` 触发 TR2/TR3 门控，**Then** 系统应检测到 plan.md 缺失，自动创建空白 plan.md 模板并继续，而不是询问"是否继续"
2. **Given** 没有 tasks.md 时运行 `vipd-implement`，**When** TR4 门控失败，**Then** 系统应记录降级原因并用传统模式继续执行

---

### 用户场景 2 — 门控内容详细记录 (P1)

作为一名项目经理，我希望每个门控检查的完整内容（检查了哪些条件、每个条件的结果、证据、时间戳）都被详细记录到结构化文件中，这样后期审计时我可以追溯每个门控决策的完整上下文。

**为什么是这个优先级**: 没有详细记录，门控就只是一个黑盒 pass/fail，丧失了审计价值。

**独立测试**: 通过全部 TR 门控后，检查 `.specify/memory/gate-status.json` 或其他存储位置是否包含每个门控的完整检查明细。

**验收场景**:

1. **Given** 一次完整的 feature 开发周期，**When** 所有门控均已执行，**Then** 门控记录文件应包含每个门控的：检查时间、检查条件列表、每项条件的结果（passed/failed/skipped）、证据描述
2. **Given** 一个门控检查包含多个 must-meet 条件，**When** 检查完成，**Then** 每个 must-meet 条件的单独结果都应在记录中体现
3. **Given** 开启了这个功能前后的项目，**When** 比较门控记录，**Then** 新记录应显著比旧记录包含更多明细

---

### 用户场景 3 — 静态 HTML 报告生成 (P2)

作为一名项目干系人，我希望查看一份格式化良好的静态 HTML 报告，以可视化的方式展示所有 TR 门控的检查历史、状态和详细内容，无需阅读原始 JSON 文件。

**为什么是这个优先级**: 可视化报告是门控记录的消费端，没有良好的呈现方式，记录的价值大打折扣。

**独立测试**: 运行报告生成命令后，打开生成的 HTML 文件，验证所有门控历史是否正确渲染。

**验收场景**:

1. **Given** 门控记录文件已包含多次门控检查历史，**When** 生成 HTML 报告，**Then** 报告应正确显示所有门控阶段（TR0-TR6）、每个门控的状态（passed/failed）、时间线和详细条件
2. **Given** 生成的 HTML 报告，**When** 在浏览器中打开，**Then** 页面应包含：门控总览仪表盘、每个门控的展开/折叠详情、色彩状态标识（绿/红/黄）
3. **Given** 生成的 HTML 报告，**When** 查看其依赖，**Then** 应为完全自包含的静态 HTML，不依赖外部 CDN、CSS 或 JS

---

### 边界情况

- **自动修复失败**: 如果 Agent 尝试自动修复门控失败（例如多次重试后仍然不通过），应有重试次数限制和最后的降级/通知机制
- **不适用自动修复**: 某些门控失败可能不应自动修复（如安全合规检查不通过），应跳过自动修复并记录原因
- **记录文件损坏**: 如果 gate-status.json 损坏或格式不正确，应优雅处理而非崩溃
- **HTML 报告数据为空**: 如果还没有门控记录，报告应显示"暂无门控记录"而不是空白页面或报错
- **大量门控记录**: 当有 50+ 条门控记录时，HTML 报告应有分页或懒加载机制，避免页面过大

## TR 门控评估 *(IPD only)*

- **适用 TR 门**: TR1 (概念), TR2/TR3 (计划), TR4/TR4A (开发), TR5 (验证)
- **风险等级**: 低 — 增强现有门控系统，不改变核心逻辑
- **门控证据要求**: 功能规格说明书、门控记录格式设计、HTML 报告模板

## 需求 *(必填)*

### 功能需求

- **FR-001**: 门控检查失败时，系统必须自动分析失败原因，而非立即询问用户"是否继续"
- **FR-002**: 系统必须根据失败原因自动执行修复操作。Agent 自主分析失败原因并决定修复方案（优先级高）；门控检查脚本可提供 `fix_hint` 作为修复参考（优先级低）。修复完成后重新检查门控。
- **FR-003**: 自动修复必须有重试上限（默认 3 次），超过后记录最终失败状态并进入降级模式
- **FR-004**: 每个门控检查必须记录详细的结构化数据到 `gate-history.jsonl`（JSON Lines 格式）：门控 ID、检查时间、检查条件列表、每项条件的结果（passed/failed/skipped）、证据描述。每次追加一行，不修改已有行。
- **FR-005**: 详细门控记录持久化存储在 `.specify/memory/gate-history.jsonl`。`.specify/memory/gate-status.json` 保持现有概览格式不变，两者互补。
- **FR-006**: 系统必须支持生成静态 HTML 报告，展示所有 TR 门控的完整历史
- **FR-007**: HTML 报告必须是完全自包含的（无外部依赖），可直接在浏览器中打开
- **FR-008**: HTML 报告必须包含：门控总览仪表盘、每个门控的展开/折叠详情、色彩状态标识
- **FR-009**: 报告生成命令 `/vipd-report` 必须可独立运行，不绑定到特定阶段，支持 --gate 过滤参数
- **FR-010**: 自动修复必须跳过"安全合规"类门控失败。识别方式：门控脚本在条件结果中标记 `category` 字段（`security`/`compliance`/`documentation`/`code` 等，优先级高）；无 `category` 字段时由 Agent 根据关键词判断（兜底）。

### 关键实体

- **门控记录 (GateRecord)**: 单次门控检查的完整数据，包含 gate ID, timestamp, conditions list, per-condition results, evidence, status
- **门控条件 (GateCondition)**: 单个 must-meet 或 should-meet 条件的名称、结果和证据
- **自动修复动作 (AutoFixAction)**: 尝试修复门控失败的操作记录，包含尝试次数、每次的结果、最终状态
- **HTML 报告 (HtmlReport)**: 生成的静态 HTML 文件，包含所有门控记录的可视化展示

## 成功标准 *(必填)*

### 可衡量的成果

- **SC-001**: 门控失败时自动修复成功率达到 80%+（对于可自动修复的类型，如缺失文件）
- **SC-002**: 从门控失败到自动修复完成的时间不超过 10 秒
- **SC-003**: 每个门控记录至少包含 5 个数据字段（gate ID, timestamp, condition list, per-condition results, evidence）
- **SC-004**: HTML 报告在所有现代浏览器中渲染正确（Chrome、Firefox、Edge）
- **SC-005**: HTML 报告从命令触发到生成完成的时间不超过 5 秒（基于 100 条门控记录）

## 假设

- 门控检查脚本（gate-check.ps1）已返回足够详细的结构化 JSON 结果，包括每项条件的单独状态
- 自动修复仅针对已知的可修复门控失败类型（文件缺失、文档不完整等），不影响需要人工判断的安全/合规检查
- HTML 报告使用内联 CSS 和 JS，不依赖外部资源
- gate-status.json 的向后兼容性 — 现有记录格式不会因新字段而破坏

## 风险登记 *(IPD only)*

| 风险 | 影响 | 可能性 | 缓解措施 |
|------|------|--------|----------|
| 自动修复误操作导致数据丢失 | 高 | 低 | 自动修复仅限于创建缺失文件/文档，不得删除或修改现有内容 |
| 门控记录文件过大 | 中 | 低 | 对历史记录设定保留期，或提供归档机制 |
| HTML 报告样式过时 | 低 | 高 | 使用简洁的基础样式，不过度设计；后期可通过 CSS 变量定制主题 |
